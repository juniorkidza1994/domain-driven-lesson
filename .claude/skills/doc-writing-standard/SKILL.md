Use this skill whenever you create or update durable project documentation.

## Core rules
- Write concise, implementation-ready Markdown.
- Optimize for a human reader who must act on the document later.
- Prefer explicit assumptions over vague placeholders.
- Keep terminology stable across related docs.
- Ask targeted questions when missing information would materially change the output.

## Traceability
- TOR defines goals, users, scope, and constraints.
- HLD translates scope into architecture and operational design.
- Backlog translates architecture into small, buildable tasks.

## Quality bar
Each meaningful doc should make these clear:
1. problem and outcomes
2. who benefits
3. scope and non-scope
4. what must be built
5. assumptions, risks, and trade-offs
6. how the team executes next
